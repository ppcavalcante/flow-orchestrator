# 0013. Pre-1.0 frozen-contract fixes ‚Äî universal `ErrRolledBack` and the `WithStore().Build()` guard

## Status

Accepted (milestone v0.13-M14, 2026-07). Records the two M14 ph62 breaking API-surface
changes (`REM-03`, `REM-04`) ‚Äî the last pre-1.0 window to get two frozen error/builder
contracts right. Both are **breaking, top-level only**, with confirmed small blast radius.

## Context

M14 is the final breaking-change window before the 1.0 freeze of all three axes (API,
`.fbs` format, interfaces). Two error/builder contracts were wrong in a way that would be
painful to freeze:

1. **A clean saga rollback was indistinguishable from a plain node failure.** When every
   compensation succeeded, `Execute` returned the bare `*ExecutionError` cause ‚Äî a caller
   had no way to answer *"did this run roll back?"* by error. Only the un-journaled
   crash-resume case surfaced the `ErrRolledBack` sentinel.
2. **`builder.WithStore(s).Build()` ran silently non-durable.** `Build()` returns a bare
   `*DAG`, and `(*DAG).Execute` has no store access ‚Äî persistence is a `*Workflow` concern,
   wired via `FromBuilder`/`Workflow.Execute`. So a caller who set a store then called
   `Build()` (instead of `FromBuilder`) got **no persistence or crash-safety while believing
   they had it.** Silent durability loss is data-loss-adjacent. (An earlier reading ‚Äî "`Build()`
   drops the store" ‚Äî was corrected by an SMTC trace: the canonical `FromBuilder` path
   propagates the store correctly; the defect was only the `Build()`-direct footgun.)

## Decision

**REM-03 ‚Äî universal `ErrRolledBack`.** A clean (fully-compensated) rollback now returns its
cause **wrapped** as `fmt.Errorf("%w: %w", ErrRolledBack, cause)`. So `errors.Is(err,
ErrRolledBack)` is **true for any rollback** (clean or the crash-un-journaled sentinel), while
`errors.As(err, &execErr)` still reaches the underlying `*ExecutionError`. A **partial**
rollback still returns `*SagaError` (unchanged). The sentinel message was generalized
`"workflow rolled back (trigger cause not journaled)"` ‚Üí `"workflow rolled back"`.

**REM-04 ‚Äî `WithStore().Build()` errors.** `Build()` now returns an `ErrValidation` error
when the builder was configured with `WithStore(...)`, directing the caller to `FromBuilder`.
The guard converts a silent durability loss into a loud, self-documenting error. Store-less
`Build()` and the store-carrying `FromBuilder` path are **unchanged**.

## Consequences

- **REM-03 breaks only bare rollback-error type-switchers** ‚Äî code doing
  `err.(*ExecutionError)` (not `errors.As`) on the rollback path. The in-repo blast radius is
  top-type-only (every in-repo saga test uses `errors.As`, which still works).
  *Migration:* detect a rollback with `errors.Is(err, ErrRolledBack)`; reach the cause with
  `errors.As(err, &ee)`.
- **REM-04 breaks only direct `builder.WithStore(s).Build()` callers** (bypassing
  `FromBuilder`). No in-repo production/example usage does this.
  *Migration:* build a store-backed run with `workflow.FromBuilder(builder)` then
  `wf.Execute(ctx)`; for a non-durable run, omit `WithStore` and use `Build()`.
- **Both preserve the moat** ‚Äî no format/determinism change; additive-only where not the two
  named breaks. The saga partition, the never-nil `ErrRolledBack` floor, and TLA/gopter
  coverage are unchanged.
